# Phase 0 — Literature & SOTA Survey: Basketball Shot Tracking from a Fixed Phone Camera

**Date:** 2026-07-13
**Scope:** Critical survey across 10 topic areas for a record-then-process half-court shot-tracking system (detect → track → homography lift → event classification), targeting eventual Core ML on-device deployment in a closed-source app.

**Evidence tags used throughout:**
- **[verified]** — the source was fetched and read during this survey (paper page, GitHub LICENSE/README, official docs, or corroborating primary snippets).
- **[recalled]** — prior knowledge, not re-verified today; treat with correspondingly lower confidence.
- **[analysis]** — our own derivation/engineering reasoning, not a citation.

Note on access conditions: `arxiv.org/abs` pages, `ultralytics.com`, `kaggle.com`, `roboflow.com`, and several publisher pages intermittently returned 403s through the research proxy. Where the primary page was blocked, verification was done via mirrors (arXiv PDF/HTML endpoints, Hugging Face paper pages, raw.githubusercontent.com LICENSE files, official GitHub READMEs) and is only tagged [verified] when a primary-equivalent source was actually read.

**Three findings that should change the plan:**
1. **The Ultralytics license risk is worse than assumed.** Ultralytics' position is that AGPL-3.0 covers not just their code but *trained weights produced with their framework*, and that even internal R&D use requires an Enterprise license unless the entire project is open-sourced [verified]. Every convenience of the YOLO ecosystem (including its excellent one-line Core ML export) is therefore off the table for this app; the Apache alternatives (D-FINE, DEIM, RF-DETR core sizes, YOLOX, RTMDet) are mature enough that this costs little accuracy, but it does cost export convenience (Section 1, 8).
2. **The heatmap-vs-bbox presumption is only half right.** The famous "TrackNetV3 ~97% vs YOLOv7 ~53%" gap is real but measured on a badminton shuttlecock (few px, per the official repo the YOLOv7 figure is 57.82%, not 53.47%) [verified]. A basketball at 20–40 px is not in that regime, and no published work shows heatmap-temporal is *necessary* at basketball scale. The temporal problem that actually matters here is rim/net occlusion, which is better addressed with an explicit ballistic-fit bridging layer than with an 8-frame heatmap network on-device (Section 2).
3. **Apple gives away more of the MVP than expected.** `VNDetectTrajectoriesRequest` (parabolic trajectory detection for stationary cameras, the WWDC20 Action & Vision demo) plus native body pose means a make/miss + shot-location MVP could plausibly ship with **zero third-party model weights** — detector training then becomes an accuracy upgrade rather than a blocker (Sections 2, 8).
4. **Freedom-to-operate needs professional review before launch.** NEX Team (HomeCourt) patent US11594029 claims essentially our planned pipeline — shooter feet at release projected through a camera/homography transform to a court location — and US11380100 covers shot-result determination (via goal-element/net movement plus trajectory) [verified at abstract/claim-1 level only]. Adjacent portfolios (ShotTracker, Shoot-A-Way, further NEX patents) make this a crowded space. Not a reason to stop Phase 0/1 R&D; it is a reason to budget for patent counsel before commercialization (Section 5).
5. **License traps extend beyond detectors.** The only basketball-specific court-calibration model (KaliCalib) is CeCILL-2.1 copyleft trained on non-commercial data; PnLCalib is GPL-2.0; T-DEED is GPL-3.0; VideoMAE weights are CC-BY-NC; the most-cited open-source make/miss repo has **no license at all** (all rights reserved by default). Each was a plausible dependency; all are now excluded or restricted to inspiration-only status (Sections 4–6).

## 1. Object Detectors — Licensing, Small Objects, Ship-Safety

**What the field does.** Real-time detection is split between the Ultralytics YOLO lineage (v8/v11/v12, now YOLO26) and an increasingly mature family of permissively licensed alternatives: the RT-DETR lineage (RT-DETR → RT-DETRv2 → D-FINE → DEIM as a training recipe → RF-DETR), plus older plain-CNN Apache detectors (YOLOX, RTMDet).

**The Ultralytics AGPL problem, confirmed.** LICENSE files for ultralytics/ultralytics, yolov12, yolov13, and YOLOE are identical AGPL-3.0 [verified]. Ultralytics' stated position: AGPL covers the training code *and the model weights produced by it*; compliance means open-sourcing the complete derivative work including the app; any use — even internal R&D — requires an Enterprise license unless the whole project is AGPL [verified via Ultralytics' own licensing docs/discussions]. App Store distribution is "distribution" under copyleft; there is no LGPL-style linking exception [analysis]. Enterprise pricing has no public sheet; one secondhand anecdote suggests ~$5k/yr for a solo developer [recalled, unreliable]. A subtle trap: Ultralytics' *bundled reimplementation* of RT-DETR is AGPL even though the original Baidu algorithm/repo is Apache-2.0 [verified] — never `pip install ultralytics` in this project.

**Small objects (~20–40 px ball).** Standard levers: higher input resolution, P2 (stride-4) detection heads, and SAHI tiled inference (MIT [verified]; +5–7 AP on aerial small-object benchmarks, more with slice-aware fine-tuning [verified]). SAHI multiplies inference cost — poor for live use but a **good fit for our record-then-process pipeline** [analysis]. Dome-DETR (arXiv 2505.05741) is a 2025 DETR specialized for tiny objects [verified existence, not evaluated]. Note that 20–40 px is COCO "small/medium," not the sub-10 px regime where per-frame detection genuinely collapses (see Section 2) [analysis].

**Recommendation.** Ship path: **RTMDet-tiny or YOLOX-S**, trained on our own data — Apache-2.0, plain CNN ops, low-risk Core ML conversion, adequate accuracy for three easy classes (ball/rim/person) at fixed framing. Accuracy upside: **D-FINE-nano or RF-DETR-nano** (best Apache small-model numbers), accepting a real Core ML engineering project (Section 8). **What would change the decision:** (a) Roboflow ships the public RF-DETR→Core ML pipeline requested in issue #318 — RF-DETR would likely become the primary; (b) Ultralytics Enterprise turns out cheap enough that one-line export + best-in-class tooling is worth paying for; (c) our ball detection recall at 240 fps input proves insufficient, elevating SAHI or a P2-head fork.

## 2. Ball Tracking — Heatmap-Temporal vs Per-Frame Detection

**What the field does.** Sports-ball tracking for tiny balls (tennis, badminton, volleyball) is dominated by the TrackNet lineage: stack N consecutive frames, regress a Gaussian heatmap of ball position, post-process into trajectories. WASB (BMVC 2023) generalized this into a multi-sport strong baseline.

**The headline claim, corrected.** The oft-quoted "TrackNetV3 ~97% vs YOLOv7 ~53%" is real but imprecise: the official repo's results table reports **97.51% vs 57.82%** (TrackNetV2: 94.98%) on the badminton Shuttlecock Trajectory Dataset [verified]; a 53.47% figure circulates in secondary write-ups without a traceable primary source [verified absence]. Two structural caveats: "accuracy" is TrackNet's custom pixel-tolerance hit criterion, not mAP (exact tolerance unconfirmed — paper PDF blocked [could not verify]); and the YOLOv7 baseline is a stock single-frame detector with no small-object tuning — no paper runs the fair comparison against a P2-head, high-res, domain-tuned YOLO [analysis].

**Basketball applicability — pressure-testing our own presumption.** The project brief presumed per-frame bbox detection is "wrong" for the ball. The evidence supports a weaker claim. For a **shuttlecock at a few pixels**, per-frame detection genuinely collapses (the 40-point gap above). A **basketball at 20–40 px** is an order of magnitude more visible; no published result shows heatmap-temporal is *necessary* at that scale, and the practitioner ecosystem (every open-source basketball make/miss repo, Section 5) uses plain per-frame detection + trajectory fitting. WASB including basketball [verified] shows heatmap machinery *works* there, not that it is *required*. The failure modes that actually threaten us are temporal — rim/net occlusion and motion blur — and both have cheaper mitigations than an 8-frame heatmap network: ballistic trajectory fitting through occlusion windows (Section 10), 240 fps capture (which slashes per-frame blur), and low-confidence detection recovery (Section 3) [analysis].

**Recommendation.** Primary: **per-frame small-object detector for the ball + explicit parabola-fit occlusion bridging**, evaluated on our own footage first. Keep **WASB (MIT)** as the offline fallback if per-frame recall near the rim proves inadequate — it is the only ship-safe, basketball-inclusive heatmap baseline with code. Use **VNDetectTrajectoriesRequest** as a zero-cost trajectory prior and MVP scaffold (Section 8). **What would change the decision:** measured ball-detection recall <~90% in the approach window on our footage, or WASB transfer experiments showing a large practical gap — in which case TrackNetV4-style motion attention (MIT) at reduced resolution becomes the on-device candidate.

## 3. Multi-Object Tracking — How Much Sophistication Is Warranted?

**What the field does.** Modern MOT is tracking-by-detection: associate per-frame detections via IoU + Kalman motion (SORT lineage), with add-ons for low-score recovery (ByteTrack), camera-motion compensation + ReID (BoT-SORT), observation-centric correction (OC-SORT), and various cue hybrids (Deep OC-SORT, Hybrid-SORT, BoostTrack). All majors are MIT-licensed [verified: ByteTrack 2110.06864, BoT-SORT 2206.14651, OC-SORT 2203.14360, Deep OC-SORT 2302.11813, Hybrid-SORT 2308.00783, BoostTrack (MVA 2024; BoostTrack++ 2408.13003)]. HOTA (arXiv 2009.07736 [verified]) decomposes evaluation into detection accuracy × association accuracy — use it over MOTA, which over-rewards detection.

**Our regime is not the benchmark regime.** MOT17/MOT20/DanceTrack feature dozens of targets and moving cameras; we have 1–3 players and a fixed camera. BoT-SORT's camera-motion compensation is moot; ReID embeddings exist to disambiguate crowds we don't have [analysis]. One relevant counterpoint: **TrackID3x3** (arXiv 2503.18282 [verified]) — fixed-camera 3×3 basketball, the closest published analog — reports its BoT-SORT-ReID baseline at only **63–66 HOTA** [verified], because same-jersey players occluding each other still cause association errors even at low counts. That argues for keeping ByteTrack's low-score box recovery (a player half-hidden behind another or the rim) while skipping ReID/GMC overhead.

**Recommendation.** **ByteTrack (MIT), or even plain Kalman+IoU**, for players; add appearance cues only if observed ID swaps actually harm shooter attribution. **Do not point a SORT-family tracker at the ball**: these trackers kill tracks after consecutive misses, which is precisely what rim/net occlusion produces; the ball gets its own detector + ballistic-bridging pipeline (Sections 2, 10). Identity persistence matters mainly for "who took the shot" in multi-player sessions; for MVP (often a single shooter) it is nearly free [analysis]. **What would change the decision:** multi-player sessions with same-color jerseys showing frequent shooter misattribution — then test Deep OC-SORT/Hybrid-SORT cue hybrids before full ReID.

## 4. Court Registration / Homography

**What the field does.** Sports-field registration research (KaliCalib, TVCalib, PnLCalib, the SoccerNet-Calibration ecosystem) solves *per-frame recalibration of moving broadcast cameras across arbitrary venues* — a much harder problem than ours. Our camera is fixed for a whole session, so one homography suffices per setup.

**Basketball court keypoint datasets** barely exist: DeepSportRadar Basketball-Instants (728 images, **CC BY-NC-SA**, Kaggle subset CC BY-NC-ND) is the only serious one [verified]; Roboflow has toy-scale sets (~29 images) [verified]. Soccer's data ecosystem has no basketball equivalent.

**Error propagation** [analysis — no published source quantifies this for wide-angle basketball capture; our own pinhole-geometry derivation, sanity-checked against KaliCalib]. Ground-position error scales roughly as h/sin²(φ) × (pixel_error/focal_px), with φ the depression angle. For an iPhone wide lens (~68° HFOV, 1920 px, f≈1420 px): good geometry (camera ~3.5 m high, ~9 m to arc, φ≈21°) puts a 2–5 px error at **~5–9 cm** on-court; bad geometry (~2 m high, 16 m to the weak-side corner, φ≈7°) at **~18–46 cm**. KaliCalib's trained model reports ~73–108 mm mean error in controlled arenas [verified] — consistent with the good-geometry band and confirming that nothing beats a few cm anyway. Shot-zone classification (bins ≥1 m wide, line width 5 cm) tolerates this easily except for feet essentially on the 3PT line — surface those as "on the line" rather than forcing a bin [analysis].

**Recommendation.** **One-time manual (tap-assisted) 4–8 point homography** at session start: key corners, FT line, 3PT arc apex, center-circle points; refine with RANSAC over detected court lines if desired. Zero license risk, works on faded driveway courts where arena-trained keypoint nets silently fail. Product guidance derived from the error model: mount high, film at max resolution, avoid grazing views of the far arc. **What would change the decision:** multi-court auto-setup at scale (user friction data showing taps are a churn point) → revisit a *self-trained* court keypoint detector using our own data engine (Section 9), not the copyleft third-party models.

## 5. Make/Miss Event Logic — Geometry vs Learned Spotting

**What open-source baselines actually do.** Every popular repo implements the same pattern: detect ball + hoop per frame (custom-trained YOLOv8), fit/smooth the ball path (linear or polynomial regression), and call a make when the trajectory passes through a hoop region while descending.

| Repo / paper | Method | Claimed accuracy | License |
|---|---|---|---|
| avishah3/AI-Basketball-Shot-Detection-Tracker | YOLOv8 ball+hoop + regression + region rule | "97% shot detection, 95% score detection" — self-reported, no test methodology [verified] | **No LICENSE file** → all rights reserved [verified] |
| kylephan5/basketball-shot-tracker | + CSRT tracker | "88%" (22/25 ad hoc) [verified] | not stated |
| nitinhemaraj / hi-tech-AI forks | same pattern | unverified | not checked |
| chonyy/AI-basketball-analysis | Faster R-CNN + OpenPose angles | none stated [verified] | MIT [verified] |
| PLOS One 2025 "Basketball detection based on YOLOv8" (BGS-YOLO) | improved YOLOv8 detection | 93.2 mAP (detection, not make/miss) [verified] | — |
| aircconline CSIT 2025 system | YOLO + pose | 89% overall (86% makes / 92% misses) [verified claim exists] | — |

Treat all accuracy claims as unaudited; none uses a public benchmark. The universal failure mode is losing the ball behind rim/net exactly at the decisive moment — the rule must integrate trajectory context before/after occlusion [verified across READMEs].

**DeepDarts as the single-camera scorekeeping analog** (arXiv 2105.09880 [verified]; wmcnally/deep-darts, no LICENSE found [could not verify]). Validates the pattern "detect a few calibration keypoints + scoring object → project to canonical plane → deterministic rule": 94.7% total-score accuracy on its easy 15 k-image set, dropping to **84.0%** on 830 images with varied camera angles [verified]. Two lessons: modest data suffices when geometry does the heavy lifting; and accuracy is viewpoint-fragile — our fixed camera avoids DeepDarts' per-throw recalibration burden entirely, but the angle-sensitivity warning transfers to miss-direction estimation (Section 10) [analysis].

**Patents (freedom-to-operate).** NEX Team (HomeCourt): **US11594029** — shot-attempt *location*: detect shooter via motion flow + ball-trajectory overlap, find feet at shot time, project foot location through the camera transform to court coordinates. Functionally our MVP location pipeline. **US11380100** — shot *result*: keys on movement of a goal element (net) plus trajectory — mechanically different from a static rim-plane crossing rule that never inspects the net [both verified at abstract/partial-claim-1 level via search-indexed text; full claim sets not read — patent pages blocked]. Related: NEX 9656142/9656143, 10748376, 11157742, 11810321, 11839805; ShotTracker and Shoot-A-Way portfolios adjacent [verified existence]. Not legal advice; a professional FTO search is a pre-launch budget line, and design-around notes (e.g., result logic that provably ignores net motion) should be kept from day one [analysis].

**Event spotting as the learned alternative.** E2E-Spot (arXiv 2207.10213, BSD-3-Clause; tennis/skating/diving/gym — no basketball) [verified] and T-DEED (arXiv 2404.05392, **GPL-3.0**; SoccerNet Ball Action Spotting 2024 winner) [verified] define frame-precise event spotting. Scale reality: the 2025 SoccerNet BAS winner (T-DEED-based) pretrained on 500+ matches and reached only ~0.6 mAP@1 [verified via secondary source]. Make/miss-as-spotting needs thousands of labeled events before it competes with geometry that needs zero event labels [analysis].

**Recommendation.** **Geometric rim-region crossing rules for MVP** — auditable, small-data, and every failure is debuggable; implement rim geometry in rim-local 3D (using calibration + known rim height) rather than raw pixels so rules transfer across camera placements. **Hybrid next step:** a small learned spotter for the *release moment* only (single class, few hundred clips) to trim clips and timestamp the feet-position read; it also de-risks the patent surface by diversifying the location method. **Longer term:** as labeled shots accumulate organically from app corrections (~1–2 k events), train an E2E-Spot-style head (BSD lineage, not GPL T-DEED) as a second opinion; promote it only if it beats geometry on held-out sessions. **What would change the decision:** if geometric rules plateau below ~95% on rim-rattle/occlusion cases despite tuning — the swish/rattle audio channel (Section 10) and a learned rim-interaction classifier become the fix rather than more geometry.

**On-device pose options.**

**Temporal models over pose.** PoseC3D (arXiv 2104.13586, Apache-2.0 via mmaction2 [verified]) is accurate but heavyweight for our clip counts; ST-GCN (arXiv 1801.07455, BSD-2-Clause [verified]) is light. Best fit: extract keypoint sequences (Vision/RTMPose), normalize to the shooter's reference frame, and train a **small custom temporal CNN / shallow ST-GCN from scratch** — pose sequences are a ~50–100× dimensionality reduction vs raw video, tractable on CPU now [analysis, grounded in verified model specs].

**Dribble detection (pull-up vs catch-and-shoot).** No peer-reviewed basketball dribble-detection paper was found [verified absence]; one hobby repo (ayushpai/AI-Basketball-Referee) tracks wrist-ball distance + ball height for double-dribble calls [verified]. Robust recipe [analysis]: over the 2–4 s pre-release window, test ball-centroid vertical oscillation (peak-finding / 1–3 Hz FFT energy, minima near floor level via homography) AND possession (ball near shooter's track); dribble ⇒ pull-up, no oscillation since catch ⇒ catch-and-shoot. This is a signal-processing feature, not a learned model — it fits the MVP data budget.

**Few-shot custom shot types (50–200 clips).** No paper tests this exact setting [verified absence]; bounding from adjacent benchmarks: X-CLIP many-way K-shot (HMDB 53→64%, UCF 76→91% as K goes 2→16) and 5-way Kinetics results (OTAM 72–84%, MRLN ~76%) [verified] imply **~80–92% for kinematically distinct classes** (pull-up vs catch-and-shoot) and **~65–80% for similar ones** (step-back vs fadeaway) with 10–40 clips/class [analysis — planning estimate, not a guarantee]. Method: **prototypical head (arXiv 1703.05175 [verified]) or linear probe on frozen embeddings**; never fine-tune a video transformer at this scale. Embedding choice is a license question: **VideoMAE/VideoMAEv2 checkpoints are CC-BY-NC-4.0** (code MIT — the code/weights license split is a trap) [verified]; **X-CLIP (microsoft/xclip-base-patch32) is MIT**, 196M params, CPU-tractable offline [verified]; InternVideo2 is 1B+ params and partly gated [verified]. **Recommendation:** pose-sequence temporal head for the built-in taxonomy; prototype head on X-CLIP (or pose-embedding prototypes) for user-defined types — prototypes make "add a custom class from a few clips" a no-retraining operation. **What would change the decision:** a permissively licensed compact video encoder distilled for sports appearing on HF (worth a periodic re-scan), or pose features proving insufficient for ball-dependent classes (e.g., floater vs runner) — then fuse trajectory features with pose.

## 8. Apple / On-Device Deployment

**`VNDetectTrajectoriesRequest`** (Vision framework, iOS 14+; showcased in the WWDC20 "Action & Vision" app) detects objects moving along **parabolic paths from a stationary camera** — precisely our capture geometry [verified via WWDC session + docs summaries; Apple's doc pages themselves are JS-rendered and blocked to fetchers]. Configuration: `trajectoryLength` (points required before a trajectory is emitted), `frameAnalysisSpacing`, `objectMinimumNormalizedRadius`/`objectMaximumNormalizedRadius` (noise filtering by size). Output: `VNTrajectoryObservation` with detected points, projected points, and parabola equation coefficients [verified]. Limits that matter [verified/analysis]: parabola-only (free flight — it will not model rim contact, bounces, or rolls); emission latency (needs N points first, so the earliest arc segment is retrospective); multiple simultaneous parabolas including artifacts (the WWDC demo itself shows a shadow generating a second trajectory) requiring size/region/direction filtering; no object identity (it doesn't know a trajectory is "the ball"). **Role: MVP scaffold and permanent cheap prior** — shot-attempt detection (an arc toward the rim region), release-point estimation (arc start), and a cross-check on the learned ball detector; the make/miss decision still needs rim-local logic near the hoop where the parabola assumption breaks [analysis].

**Core ML conversion reality.** coremltools 9.0 (Nov 2025) [verified]: mature for CNNs — conv/pool/concat convert cleanly with fp16/int8 PTQ and palettization for the Neural Engine [verified]. Measured reference points: YOLO26n detect and pose each **3.8 ms on iPhone 17 Pro ANE (INT8)** vs 9–12 ms CPU [verified via Ultralytics benchmark table — useful as an upper-bound proxy for what a small Apache CNN of similar size will do]. **DETR-family conversion is genuinely unsolved off-the-shelf**: deformable attention's `grid_sample` has open coremltools issues (fp16 accuracy divergence #2068, rank-5 unsupported #2284) [verified]; the HF forum shows no successful RT-DETR conversion (dtype-mismatch failure via the AGPL exporter; no resolution posted) [verified]; RF-DETR has an open feature request for a public Core ML pipeline (#318) with a community attempt blocked on bicubic upsample [verified]; RT-DETRv2's own abstract advertises a discrete-sampling op to escape `grid_sample` for deployment [verified] — the escape hatch exists but nobody has published a completed Core ML port [verified absence]. Historic `scaled_dot_product_attention` failures (#1823) may be fixed in 9.0 [could not verify].

**Pipeline sizing** [analysis]: record-then-process removes the real-time constraint — a 30 s clip at 240 fps is 7,200 frames, but detection can run on a decimated stream (e.g., 60 fps equivalent) with full-rate analysis only in the ±0.5 s rim-approach window; at ~4–10 ms/frame for a small INT8 CNN on ANE, a session processes in well under clip duration. Create ML Action Classification (pose-window classifier over Vision keypoints) is a legitimate zero-code baseline for shot-type before custom temporal heads [recalled — not re-verified today].

**Recommendation.** MVP on **Vision framework primitives** (trajectories + body pose + manual homography + geometric rules); first trained model = small Apache CNN detector converted via coremltools with INT8 PTQ. Treat any DETR-family on-device plan as a research spike with explicit exit criteria. **What would change the decision:** coremltools or Roboflow shipping a blessed DETR/RF-DETR export path (watch #318), or Apple expanding Vision's sports primitives further.

## 10. Novel Angles — Audio and Single-Camera Physics

**Audio for make-quality (swish vs rattle vs clank). Feasibility: MEDIUM, and a genuine gap.** Targeted searching found **no academic work on basketball shot-outcome classification from audio** [verified absence — searched, not merely unrecalled]. Adjacent evidence says the mechanism is sound: table-tennis spin/racket classification from bounce sound via CNN (arXiv 2409.11760, Sony AI) [verified]; tennis ball-hit acoustic event detection (ACM MMSports) and a tennis acoustic timing pilot at >95% event detection [verified]. Swish (broadband net whoosh), rim rattle (repeated metallic transients), clank (single hard transient), and backboard (low resonance) are acoustically well-separated classes by construction [analysis]. Plan: log-mel windows around the vision-determined rim-arrival time → frozen embedding (YAMNet for on-device simplicity; PANNs/BEATs/AST offline — BEATs weights MIT [recalled]) → small classifier head; a few hundred self-labeled impact clips across ≥3 gyms/rims to control reverb/hardware variance; FSD50K (arXiv 2010.00475, CC-licensed [verified]) for optional pretraining. Audio is a *confirmatory* channel gated by vision timing — alignment needs vision to know when the ball reached the rim; it directly attacks the two hardest vision cases (swish-vs-rattle make quality, rim-occlusion ambiguity). Risks: outdoor wind/traffic noise, multi-ball gyms, phone AGC; distance-to-rim attenuation at half-court range [analysis].

**Physics / ballistics from one fixed camera. Feasibility: MEDIUM, with an honest degenerate case.** Post-release flight is a parabola in a vertical plane; the observables are the 2D image track plus two anchors — release region and the calibrated 3D rim position — plus the ball's known diameter (~24 cm) as a per-frame depth-from-size cue. Prior art: physics-based monocular basketball 3D trajectory reconstruction exists (Chen et al. 2009, JVCIR; IEEE IECON 2023 monocular ballistic reconstruction) [verified existence, abstracts only]; DeepSportRadar's ball-3D-localization task operationalizes diameter-based depth [verified]; "Where Is The Ball" (arXiv 2506.05763, CVPR'25 workshop) learns camera-independent 3D ball localization from synthetic data [verified existence; code status unclear]. Honest limits [analysis]: (a) depth-from-size at 20–40 px is noisy — a ±1 px diameter error is ±3–5% depth error (≈±0.4–0.7 m at 14 m), so per-frame size cues must be filtered along the whole arc, and motion blur inflates apparent diameter (240 fps capture directly helps); (b) accuracy is azimuth-dependent — side-on arcs are well-observed, near-end-on arcs (shooter shooting almost along the optical axis) collapse the short/long axis into the depth direction; (c) spin/drift breaks the single-vertical-plane assumption modestly. Consequences: **left/right miss classification is robust** (mostly image-plane geometry near the rim); **short/long is the hard axis** whose reliability varies by shot position for any single fixed camera — the app should either constrain camera placement (elevated, ~30–60° off the shooting lane) or report short/long with confidence gating. Occlusion bridging via parabola fit is the same machinery and is low-risk [analysis]. Miss-type ground truth is cheap to self-label (each miss is visually obvious), so this tier can be validated empirically before promising it to users.

## Risks & Open Questions

1. **Patent exposure (highest non-technical risk).** US11594029 reads directly on feet-at-release→homography shot location; adjacent NEX/ShotTracker portfolio is crowded. Action: professional FTO search before commercialization; document design-arounds early. [verified at abstract level]
2. **DETR-on-ANE is unproven.** If small-CNN accuracy on the 20–40 px ball disappoints AND DETR conversion stays hard, the detector plan has no third option beyond paying Ultralytics. Mitigation: early conversion spike; watch RF-DETR #318 and coremltools releases.
3. **No public dataset matches our domain.** Fixed wide-angle amateur half-court footage is unrepresented; transfer sets are broadcast/arena. The self-collection data engine is on the critical path, not a nice-to-have.
4. **Unverified accuracy folklore.** The 95–97% claims in open-source make/miss repos are self-reported without methodology; DeepDarts' 94.7→84.0% angle-transfer drop is the more honest prior. Expect a real evaluation harness (per-session held-out) to be sobering.
5. **Ball recall near the rim is the pivotal unknown.** The heatmap-vs-bbox decision was made on shuttlecock evidence extrapolated to basketballs; our recommendation (bbox + ballistic bridging) must be validated in week one on our own footage. Exit criterion defined in Section 2.
6. **License hygiene is a process, not a one-time check.** Weights licenses diverge from code licenses (VideoMAE, YOLO-NAS, Qualcomm RTMPose exports); Roboflow per-project badges were not all individually confirmable today (site 403s). Re-verify every dependency's LICENSE file at integration time.
7. **Open questions:** zero-shot Grounding DINO/OWLv2 recall on "basketball"/"rim" (unbenchmarked); TrackNetV5 (2512.02789) contents (unread); coremltools 9 SDPA status; VNDetectTrajectories behavior at 240 fps slow-mo timebases; Create ML Action Classification ceiling; audio separability at half-court distance outdoors; whether short/long confidence gating is acceptable UX.
